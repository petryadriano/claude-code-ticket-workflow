> **$WORKSPACE_ROOT** = parent of `.claude/` = current workspace root (e.g. `C:/source/ws1`, `C:/source/ws2`). All paths below resolve from this root. The product repos (`web`, `api`, `auth`, …) live directly under it.

## What this skill is (and is not)

`start-stack` is the **stack bring-up layer** — it gets a runnable stack up so `verify-ticket` has
something to run its specs against and so you can debug by hand. It is the mechanism behind the "ensure
the stack is up" precondition that `verify-ticket` (Step 2) and `implement-ticket`'s visual checks
(Step 6) assume — the design doc flagged bring-up as the biggest open gap
(`docs/automated-verification-design.md`, Phase 1), and this fills it.

It is **not** the evidence layer. It brings the stack up and reports URLs; it does **not** run specs,
collect evidence, or judge ACs — that stays in `verify-ticket`.

> **The one rule that decides correctness, whichever bring-up you choose: the UI must talk to your
> LOCAL API** (your working-tree code) — *not* a deployed env's backend. Deployed QA runs merged code
> only (that's the `qaApi`/regression path). A backend change verified against deployed backends proves
> nothing. The two bring-ups below (Docker / local processes) are equivalent **as long as the UI points
> at the local API**; the e2e harness is target-agnostic (`E2E_UI_URL` + `E2E_UI_PATH` select which one
> the UI runtime specs drive — `fixtures/ui`).

> **Forward-compat:** the design doc's end state is per-service `run-*` skills generated by
> `/run-skill-generator` (deterministic launch + health/port polling), with this skill as the thin
> orchestrator over them. Those don't exist yet — until they do, this skill uses the recipes below
> directly. When a `run-<service>` skill is present for a service, prefer invoking it.

## Required disciplines (Superpowers substrate)

- **REQUIRED: superpowers:systematic-debugging** — when a service won't come up (container exits, port
  never answers, the UI can't reach the API), read the **complete** error / container log and
  root-cause it before retrying. Never loop `up` blindly or report a false "ready".
- **REQUIRED: superpowers:verification-before-completion** — never claim a surface is "up" without a
  green readiness probe (Step 4). A built container / started process is not a serving one.

## Step 1 — Decide what to bring up (don't start everything)

If a `<ticket>` was given, read `$WORKSPACE_ROOT/.claude/tickets/<ticket>.json` for `plan.repos`, the AC
types, and `test_scope` (target environment + impersonation target). Bring up only what the ticket needs, and
**name what you are deliberately NOT starting**:

- **Backend repos under test** (`api`, `auth`, … in `plan.repos`) → must run
  **locally** so working-tree behavior is exercised.
- **auth** → bring it up locally whenever anything authenticates (it does).
- **UI or visual AC** → the UI, **pointed at the local API** —
  served either by the Docker web host or the dev server (Step 3).
- **No ticket given** → bring up the stack for ad-hoc debugging.

## Step 2 — Prerequisites (assume ready; surface only on failure)

Assume the developer's environment is already set up — **network connectivity and cloud session valid** (it
normally is). **Do not prescribe a credentials refresh / "connect the network" upfront.** Surface them only as the
*cause* when something actually fails: a cloud-config read failing → refresh credentials; the target
DB host / QA domain not resolving (a probe can't connect) → network connectivity. The one thing worth a proactive check
is the **toolchain**: Docker (Option A) or the runtime SDK + Node (Option B), per the root CLAUDE.md.

## Step 3 — Bring the stack up (pick A or B; both must point the UI at the local API)

### Option A — Docker compose (one command; production-fidelity build)

One `docker compose up` builds + serves the web host and the UI (the integrated, built app — the
closest thing to how it ships). Per the root CLAUDE.md docker convention every repo ships a `local`
compose service:

```bash
cd "$WORKSPACE_ROOT/web" && docker compose --file docker-compose.yml up -d --build local
```

To make this a real **verification** surface (not just frontend poking), wire it correctly:
- **Point the web host at your LOCAL API.** From inside the container the host API is `host.docker.internal`
  (e.g. `https://host.docker.internal:<api-port>`) — set the API base URL the container uses to that, or run
  the API as another service on the same compose network. Only when the web host hits your local API does a
  working-tree **backend** change get exercised. (For a **frontend-only** ticket, hitting a deployed
  env's backend is acceptable — the UI in the `--build local` image is still your working-tree code.)
- If the config reads cloud credentials, give the container creds (mount the host cloud-config dir or pass
  exported credentials as env); set a local-only
  a local-only domain-override setting (= `<your-domain>`) in your per-environment config file for environment
  resolution. **That override is a debug-only edit — do NOT commit it / add it to `files_changed`.**
- **Point the harness at it:** `E2E_UI_URL=https://localhost:<hostPort>` and
  `E2E_UI_PATH=/{environment}/<ui-route>/{recordId}/{versionId}` in `e2e/.env`.

> **Confirm in the web repo (can't be assumed):** the exact `local` compose service + host port, how
> its API base URL is configured (so you can point it at `host.docker.internal`), whether the
> host-served UI accepts the base64 `?auth=` param (it's SPA-level, so likely yes), and the
> environment-prefixed record URL shape (→ `E2E_UI_PATH`). Surface these to the user; don't guess.

### Option B — Local processes (proven recipe; fast iteration)

This is what worked end-to-end against a real QA environment — use it as the concrete default:

1. **Local API** — run from the PROJECT dir (so it loads your local config file — which holds any
   secret keys; without it startup may fail):
   ```bash
   APP_ENV=<qa-env> APP_URL=https://localhost:<api-port> <run command for your API>   # started from the project dir
   ```
   In `<qa-env>` it loads QA config and resolves the target environment from the request
   token. **No local auth needed** — `E2E_AUTH_URL` points at the deployed QA auth (the token lands
   in the shared auth store).
   - *Gotcha (secure workspaces):* a freshly-built binary can be EDR/AV-blocked ("Access is denied").
     Unblock it once, **or** run it via your runtime's host — but still **from the project dir**
     (launching from elsewhere can resolve the wrong working directory, so your local config won't load).
2. **UI dev server (for UI specs)** — `npm install` once, then run the **dev** server (NOT
   `npm start`, which is a *production* serve that 404s on a missing `dist/`):
   ```bash
   cd "$WORKSPACE_ROOT/web/<ui>" && AUTOMATED_TEST=true \
     AUTH_SERVER=https://auth.qa.example.com npm run dev      # EXAMPLE — replace with your app's specifics
   ```
   - **🔴 POINT THE UI AT YOUR LOCAL API — the one rule that makes this a working-tree test.** The UI
     resolves its DATA API host from **`ENV_MAP[env].hostname` in `web/<ui>/client/utils.js`** *(EXAMPLE — replace with your app's specifics)*
     (`loadApiData`). The e2e auth param uses `env: 'QA_AWS'`, whose hostname defaults to the **DEPLOYED**
     `https://api.qa.example.com` (no unmerged code). **`API_SERVER=…` on `npm run dev` does NOT redirect
     this.** So you MUST override it (debug-only; dev server hot-reloads):
     ```
     // client/utils.js → ENV_MAP.QA_AWS.hostname:   // EXAMPLE — replace with your app's specifics
     hostname: 'https://localhost:<api-port>',   // was https://api.qa.example.com
     ```
     Without this, the UI reads ALL data from deployed QA and your working-tree backend is never exercised
     — a backend-populated value (e.g. a new column) renders BLANK while the API returns it,
     and pre-existing columns still show, masking the problem. **This is a debug-only edit — do NOT commit it
     / add it to `files_changed`.** Confirm in Step 4 that a record request actually hits `localhost:<api-port>`.
   - **`AUTOMATED_TEST=true`** *(EXAMPLE — replace with your app's specifics)* makes the UI emit semantic `data-test` attributes (the plain dev build emits
     NONE). **`AUTH_SERVER`** points auth at the deployed QA auth (no local auth needed). Serves on
     `http://localhost:3000` (webpack compile ~3 min).
   - **Reading the grid:** prefer **`readGrid`** *(EXAMPLE — replace with your app's specifics)* (the proven READ_CONSOLE technique — reads the SELECTED
     row's console row `ConsoleSelection-module-root`, header↔value by left-x, keyed by column LABEL). It
     reads backend-derived values correctly. (`readConsoleGrid`'s data-test base
     cells read blank for those — it's only useful for non-derived columns.)
   - Harness default: `E2E_UI_URL` unset → `http://localhost:3000`, `E2E_UI_PATH` → `/{recordId}/{versionId}`.
   - The asserted columns must be **displayed in the record's grid view** (column config) — a per-ticket data precondition.

> Whichever option: the e2e specs don't change — they read `E2E_UI_URL`/`E2E_UI_PATH`. Switching targets
> is a `.env` change, not a code change.

## Step 4 — Probe readiness and report URLs

Verify, don't assume (the verification-before-completion instance):

- **Probe each surface** you started:
  `curl -sk -o /dev/null -w "%{http_code}" --max-time 5 https://localhost:<port>/` (local API, auth, web host,
  or UI dev server `3000`, … per the root CLAUDE.md port
  table). A started container/process that doesn't answer is **not** ready — root-cause via
  systematic-debugging (read its log), don't report green.
- **Warm the UI before handoff.** A cold dev-server compiles on the FIRST navigation, so an un-warmed UI
  can make the first `page.goto` in `verify-ticket` time out (~90s) and look like a stack failure. After it
  serves, hit a record URL once (`curl` or a throwaway navigation) so the first real spec navigation is fast.
- **Confirm the UI reaches the local API** (the rule from the top): load a record and check a request goes
  to your local API, not a deployed host. If it doesn't, the stack is mis-wired — fix that before
  handing off, or `verify-ticket` will test the wrong code.
- **If a surface is already serving, reuse it** — don't restart a healthy one.
- **Report the URLs** the user (and `verify-ticket`) will use, derived from `test_scope`:
  - **UI (what the specs drive):** `${E2E_UI_URL}${E2E_UI_PATH}` for the record — pointed at your
    local API.
  - **QA comparison:** `https://app.qa.example.com/<environment>/<ui-route>/<record>` — the same flow on
    deployed QA, for the human's side-by-side compare.

Then tell the user:

```
✓ Stack up for <PROJ-XXX | ad-hoc>  [bring-up: Docker | local processes]:
  Local API (working-tree):    https://localhost:<api-port>   <or "not needed — UI-only ticket">
  Local auth:                  https://localhost:<auth-port>
  UI:                          <E2E_UI_URL + record path>   (verified → local API)
  Not started:                 <list what you deliberately skipped>

  Compare on QA: https://app.qa.example.com/<environment>/<ui-route>/<record>

Next: /verify-ticket <PROJ-XXX> to run the specs + collect evidence against this stack.
```

If anything failed to come up, lead with it and the root cause — never bury it under what did start.
